# The opt-in live integration tier

Tutti's required CI gate is hermetic: no `claude`, no `gh`, no network. Everything
process- or network-adjacent is covered instead by pure unit tests over its parsing
and prompt-building seams (stream-json parsing, gh-json parsing, artifact reading,
prompt construction, PID locking).

The one place that cannot be fully covered by a pure test is whether `claude -p`
actually behaves the way `ClaudeBackend` expects when invoked against a real,
authenticated `claude` binary. That is covered by a separate, opt-in "live" tier.

## What it is

`crates/tutti-backend-claude/tests/live.rs` is gated behind the `live` cargo feature
and marked `#[ignore]`. It does not compile at all unless `--features live` is passed,
so it can never leak into the required gate. When run, it spawns a real `claude`
process against a throwaway `AgentTask` and asserts the run never panics: it must
land on either `AgentStatus::ReadyToShip` or `AgentStatus::Blocked`.

## Running it

Prerequisites:
- An authenticated `claude` CLI on `PATH` (run `claude --help` to confirm it is
  usable and check current flags before trusting the backend's invocation).
- For the forge-side live tests envisioned by this tier in a future slice, a
  throwaway GitHub repo configured with the `status:*` labels the engine expects.

```bash
cargo test -p tutti-backend-claude --features live -- --ignored
```

## Rules

- Never add this tier, or the `live` feature, to any required CI workflow. It costs
  real tokens, needs a real credentialed `claude`, and is not deterministic enough
  to gate merges on.
- If a change to `ClaudeBackend` needs verifying against real `claude` behavior
  (invocation flags, output shape), run this tier locally before merging, but the
  required gate result is what actually blocks the PR.
